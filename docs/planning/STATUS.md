- 2026-07-23: **Phase 5 — OPA policy gate complete** (FHE still not started; see above). Added an `opa` service to `docker-compose.yml` (`openpolicyagent/opa:1.4.2-static`, mounting `policy-plane/rego/` directly — no manual policy-load step, same pattern as Fuseki's ontology mount). `policy-plane/rego/schema_mutation.rego` treats the shim's alias-learning (self-healing) as a schema mutation subject to Zero-Trust policy rather than something that applies unconditionally once the fuzzy-matcher finds a unique candidate: non-identity fields (e.g. `trackId`) heal for any source, but `userId` — flagged as identity-bearing/PII-adjacent — only heals for a source on the `trusted_sources` allowlist.
  New `opa.rs` module (mirrors `shacl.rs`'s HTTP client pattern): `OpaClient::Remote` posts `{"input": {source, field, raw_key}}` to OPA's `/v1/data/.../allow` endpoint; `OpaClient::Disabled` (used when `policy_plane` is omitted from config) always allows, so local dev doesn't require standing up OPA to see self-healing work. **Fail-closed by design:** an unreachable OPA denies rather than defaulting to allow — unlike a Fuseki outage (which just leaves one event unhealed for now), a security gate that fails open on outage would defeat the point of having it. `validate::process` now calls `opa.allow_mutation()` before `shim.learn_alias()`, gated on a genuine per-event identity rather than a diagnostic label — this surfaced a real bug during implementation: `RawEvent.source` is a *log* label (`amqp:<queue>:<tag>` or a full filesystem path — the AMQP delivery tag isn't even stable across redeliveries, and the file path is verbose and arbitrary), unfit to be a trust boundary. Added a new `RawEvent.origin` field instead: the filename stem for file-watch, the AMQP `app_id` property (the property actually meant for publisher identity) for RabbitMQ, falling back to `"unknown-app"` if absent.
  Verified live end-to-end (not just unit tests): dropped a file-watch event named `untrusted-source.json` renaming `userId` to `user_id` — denied, confirmed `user_id` stayed `NULL` in DuckDB — then a `partner-feed.json` event with the same rename — allowed and healed. Also added 2 new unit tests (`opa.rs`'s `disabled_client_always_allows`/`unreachable_opa_denies_fail_closed`, `validate.rs`'s `unreachable_opa_leaves_field_unhealed`) plus 4 live-OPA integration tests (`#[ignore]`, run with `cargo test -- --ignored` after `docker compose up -d opa`) covering all three policy branches end-to-end through `validate::process`. All 28 tests pass (21 unit + 7 live-service, up from 21+3 — the +3 being the pre-existing live-Fuseki tests).
  **Accepted scope boundary, found and documented rather than fixed:** alias learning is global and the `streaming_events` table is a view that heals retroactively (by design since Phase 3 — see the Phase 3 shim note above), so OPA only gates *who may teach* the fabric a new alias, not *who benefits* from one once it's learned. A source denied for attempting `user_id -> userId` still gets that field healed for free the moment any trusted source teaches the same rename, since the view re-heals every row sharing that raw key once learned, not just the trusted source's own rows. Treated as consistent with the shim's existing model (a legitimate rename is a fabric-wide contract fact, not a per-tenant one) rather than a bug — same posture as Phase 3's D1 finding.
- 2026-07-23: **Phase 5 — FHE-encrypted aggregation complete.** Scoped "encrypted aggregation on user IDs" concretely first: encrypts the numeric usage value (`msPlayed`) via OpenFHE (BFV scheme, coefficient-packed single-integer plaintexts — chosen over the batched/NTT `MakePackedPlaintext` encoding specifically to avoid needing an NTT-compatible prime plaintext modulus), homomorphically sums a user's ciphertexts, and decrypts only the final aggregate; `userId` itself stays a plaintext grouping key (like a DB partition key), not something FHE-protected — the alternative (protecting the identity field itself) would need PSI-style techniques, out of scope for a baseline.
  Built `privacy-plane/fhe/` — a small FastAPI service (`app.py` + `Dockerfile`) added to `docker-compose.yml` as the `fhe` service. Confirmed empirically (openfhe-python has no in-memory serialization API, only file-based) rather than assumed: verified the exact API surface — `MakeCoefPackedPlaintext`/`GetCoefPackedValue`, `SerializeToFile`/`DeserializeCiphertext` round-tripped through a temp file — by running a standalone script inside the built container before wiring up the real service, since guessing at binding signatures from docs alone had already proven unreliable (the two official example scripts even disagree on `Decrypt`'s argument order). Also had to add `libgomp1` to the image after the bare `openfhe` wheel failed with a missing `libgomp.so.1` at import time — the PyPI package is a `py3-none-any` wheel that bundles a compiled `.so` linked against Ubuntu 24.04's system libraries specifically (not a portable manylinux wheel), matching the ARCHITECTURE_DECISIONS.md note about why this had to be Ubuntu-based, not `python:slim`.
  New `fhe.rs` Rust module (mirrors `opa.rs`'s HTTP client pattern): `encrypt()` is a privacy feature, not a security gate — unlike OPA's fail-closed policy check, a failed encryption just skips that event's ciphertext-at-rest copy (logged, non-fatal) rather than blocking processing, since `msPlayed` is optional in the SHACL contract to begin with. New `encrypted_metrics` DuckDB table (`shim.rs`) stores ciphertexts keyed by `event_id`/`user_id`; `validate::process` encrypts whenever both `userId` and `msPlayed` resolved, independent of full SHACL conformance. Added a one-shot orchestrator CLI mode, `--aggregate-user <id>`, that fetches a user's ciphertexts, calls the FHE service's `/aggregate`, and prints the decrypted total — making "run an encrypted aggregate query" an actual capability rather than a one-off curl/duckdb script.
  Verified live end-to-end: dropped three events for the same user (`msPlayed` 180000/210000/195000) via file-watch, confirmed `encrypted_metrics` held only opaque ~88KB ciphertext blobs (no plaintext `msPlayed` anywhere but the raw JSON audit log), then ran `--aggregate-user` and got back exactly 585000 — matching the plaintext sum, without the aggregate step ever decrypting an individual event. 35 tests pass (25 unit + 10 live-service integration, up from 21+7 — the +7 being OPA's live tests from the note above), including a live round-trip test asserting the FHE service's own decrypted sum matches an independently-computed plaintext sum.
  **Accepted scope boundaries, documented rather than fixed (baseline demo, not a production KMS):** (1) the FHE service holds one in-process keypair for its whole lifetime, generated fresh on every container start — a restart makes every previously-encrypted ciphertext permanently undecryptable under the new key (silently wrong, not an error, since deserialization succeeds but decryption under the wrong key just produces garbage). No key persistence/rotation exists. (2) `PLAINTEXT_MODULUS = 100_000_000` bounds the maximum summable total — BFV arithmetic is modular, so exceeding it wraps silently rather than erroring. Both are called out in `privacy-plane/fhe/app.py` and would need real key management/parameter sizing before this became more than a demo.
